continuous integration

Use contradish as a CI gate

Block deploys when your LLM app starts contradicting itself. Add contradish to your pipeline and catch reasoning regressions before they reach production.

how it works

01
Define your test cases
Write one test case per behavior you care about. Each one is a question or prompt your app should answer consistently.
02
contradish generates variants
Contradish rewrites each input into multiple semantically equivalent paraphrases and runs your app against all of them.
03
Fails the build on contradiction
If your app gives contradictory answers across variants, the suite exits with code 1 and the pipeline stops.

Why this matters. Standard unit tests check that your app returns the right answer to one specific question. They don't catch the case where your app says yes to one phrasing and no to another. Contradish catches that.

basic setup

Add this to any CI script. The suite exits with code 1 if any test fails, which stops your pipeline.

python — ci_check.py
# ci_check.py
import sys
from contradish import Suite, TestCase

# Your LLM app — any callable that takes a str and returns a str
def my_app(question: str) -> str:
    return your_llm_or_agent(question)

suite = Suite(app=my_app)

suite.add(TestCase(name="refund policy", input="Can I get a refund after 45 days?"))
suite.add(TestCase(name="shipping time", input="How long does standard shipping take?"))
suite.add(TestCase(name="return window", input="What is your return policy?"))

report = suite.run(paraphrases=5, verbose=False)

if report.failed:
    print(f"{len(report.failed)} test(s) failed — reasoning instability detected")
    for r in report.failed:
        print(f" {r.test_case.name}: consistency={r.consistency_score:.2f}")
    sys.exit(1)

print("All tests passed.")

github actions

Drop this workflow file into your repo. It runs your contradish suite on every push and pull request.

.github/workflows/contradish.yml
name: contradish

on:
  push:
    branches: [main]
  pull_request:

jobs:
  reasoning-stability:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install contradish
        run: pip install "contradish[anthropic]"

      - name: Run reasoning stability checks
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: python ci_check.py

Store your key as a secret. In your GitHub repo go to Settings → Secrets and variables → Actions → New repository secret. Add ANTHROPIC_API_KEY or OPENAI_API_KEY as the name.

yaml config

Keep your test cases in a YAML file and run them from the CLI. Useful when you want to manage evals separately from your application code.

evals.yaml
test_cases:
  - name: refund policy
    input: Can I get a refund after 45 days?

  - name: return window
    input: How long do I have to return something?

  - name: shipping estimate
    input: When will my order arrive?
    expected_traits:
      - should give a timeframe
      - should not promise a specific date
terminal — CLI usage
# run from a YAML file
contradish run evals.yaml --app mymodule:my_app_function

# with more paraphrases for a thorough check
contradish run evals.yaml --app mymodule:my_app --paraphrases 8

configuring thresholds

Override the default pass/fail thresholds to match your application's tolerance for inconsistency.

python
suite = Suite(app=my_app)

# default: consistency=0.70, contradiction_max=0.30
# tighten for high-stakes apps like legal or medical
suite.thresholds(
    consistency=0.90,      # fail if consistency drops below 0.90
    contradiction_max=0.10, # fail if contradiction rate exceeds 0.10
)
default thresholds
consistency ≥ 0.70
contradiction ≤ 0.30
good for most apps
strict thresholds
consistency ≥ 0.90
contradiction ≤ 0.10
high-stakes apps

what it catches

✓
Prompt regressions Your system prompt changed and the model now answers a policy question differently depending on how it's phrased.
✓
Model version drift You upgraded to a new model version and it's less consistent on edge-case phrasings than the previous one.
✓
RAG retrieval inconsistency Your retrieval layer returns different chunks for semantically equivalent queries, causing the model to give conflicting answers.
✓
Context window sensitivity Small changes in the conversation history cause the model to flip its position on a factual or policy question.
